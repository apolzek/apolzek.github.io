**saving your time**: *a hands-on day 1 with eBPF — what it is, why it exists, and a small hello-world that traces every `execve` on the box*

## What is eBPF, really

eBPF (extended Berkeley Packet Filter) lets you run sandboxed programs inside the Linux kernel without writing kernel modules, without recompiling the kernel, and without rebooting. The kernel verifies the program before it runs, attaches it to a hook (a syscall, a tracepoint, a network event, a kprobe), and executes it in-kernel whenever that hook fires.

In one sentence: *eBPF is a safe, programmable way to ask the kernel "tell me when X happens, and run this little function when it does."*

The interesting part is not the bytecode itself, it's the ecosystem around it: maps to share data with user space, helpers to read kernel structures, and tooling that hides the C/clang/LLVM dance behind something more pleasant.

## Why people care

- **Observability without agents on the hot path.** You can trace syscalls, file opens, TCP connects, DNS queries, and aggregate them in-kernel — the user-space process only reads the summary.
- **Networking.** Cilium, kube-proxy replacements, load balancing, and policy enforcement all run as eBPF programs attached to XDP or `tc`.
- **Security.** Tools like Tetragon and Falco use eBPF to enforce policy and detect anomalies based on real kernel events, not log files.

The trade-off: every program is verified by the kernel verifier. Loops, unbounded memory access, and "trust me" pointers will be rejected. This is a feature, not a bug.

## Picking a path: bcc, bpftrace, libbpf, or CO-RE

You will see four names a lot:

| Tool | What it is | Good for |
| ----- | ---------- | -------- |
| **bcc** | Python/C frontend, compiles programs on the target host | Quick prototyping; lots of existing scripts (`opensnoop`, `execsnoop`) |
| **bpftrace** | High-level scripting language (think awk for the kernel) | One-liners, ad-hoc tracing during incidents |
| **libbpf** | C library, programs compiled ahead of time | Production tools |
| **CO-RE** | Compile Once, Run Everywhere — uses BTF to relocate field offsets at load time | Production tools that ship one binary across kernel versions |

For day 1, start with **bpftrace**. The feedback loop is seconds, not minutes.

## Setup

On a recent Ubuntu / Debian / Fedora / Arch you only need the package:

```sh
# Ubuntu / Debian
sudo apt install bpftrace

# Fedora
sudo dnf install bpftrace

# Arch
sudo pacman -S bpftrace
```

Sanity check that your kernel has BTF and the right config:

```sh
ls /sys/kernel/btf/vmlinux && echo "BTF: ok"
zgrep CONFIG_BPF /proc/config.gz 2>/dev/null || \
  grep CONFIG_BPF /boot/config-$(uname -r)
```

If `/sys/kernel/btf/vmlinux` exists, you have BTF, which means modern CO-RE tools will work without kernel headers installed.

## Hello world: trace every new process

This is the eBPF equivalent of `print("hello")`. Save nothing — just paste this in a terminal:

```sh
sudo bpftrace -e 'tracepoint:syscalls:sys_enter_execve {
  printf("%-6d %-16s %s\n", pid, comm, str(args->filename));
}'
```

Now open another terminal and run anything (`ls`, `git status`, `curl example.com`). You will see every process exec'd on the box, in real time, with no agent, no instrumentation, and no restart.

What is happening:

1. `tracepoint:syscalls:sys_enter_execve` — the hook. The kernel will run your program every time a process calls `execve`.
2. `args->filename` — the syscall argument, read via the tracepoint's stable ABI.
3. `printf` — the bpftrace helper pushes the formatted line to user space through a perf ring buffer.

## A slightly more useful one: count syscalls per process

```sh
sudo bpftrace -e 'tracepoint:raw_syscalls:sys_enter {
  @[comm] = count();
}'
```

Hit `Ctrl+C` after a few seconds. bpftrace prints the map, which is a histogram of syscall counts grouped by process name. The aggregation happens **inside the kernel** — user space only receives the final map.

This is the trick that makes eBPF cheap: hot paths stay in the kernel, only summaries cross the boundary.

## What broke for me on day 1

- **Verifier rejection on a loop.** I wrote a `while` and got `BPF program is too large`. The verifier has bounded execution; use `unroll` or rewrite as a tail call.
- **`Failed to attach probe`** — the tracepoint name was wrong. `sudo bpftrace -l 'tracepoint:syscalls:*'` lists everything available.
- **Empty output on a container host.** The probe was attached on the host's kernel (correct) but the workload was inside a container with a different PID namespace. `comm` and `pid` are the host's view; map them with `pid` → cgroup if you need container context.
